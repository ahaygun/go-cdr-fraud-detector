KEDA autoscaling of the fraud consumer on Kafka consumer-group lag
(local kind cluster; load applied with `make k8s-load`).

# Baseline — 1 replica, no lag:
$ kubectl get hpa keda-hpa-fraud
NAME             REFERENCE          TARGETS   MINPODS   MAXPODS   REPLICAS
keda-hpa-fraud   Deployment/fraud   0/100     1         3         1

# Under load — lag builds past the threshold, KEDA scales fraud 1 -> 3:
$ kubectl get hpa keda-hpa-fraud
NAME             REFERENCE          TARGETS     MINPODS   MAXPODS   REPLICAS
keda-hpa-fraud   Deployment/fraud   .../100     1         3         3

$ kubectl get pods -l app=fraud
NAME                     READY   STATUS    RESTARTS   AGE
fraud-79bc659c8d-g7l8t   1/1     Running   0          40s
fraud-79bc659c8d-jt9c2   1/1     Running   0          40s
fraud-79bc659c8d-r76m7   1/1     Running   0          40s

$ kubectl get scaledobject fraud
NAME    SCALETARGETNAME   MIN   MAX   READY   TRIGGERS
fraud   fraud             1     3     True    kafka

Because cdr.raw is partitioned by caller_msisdn, the scaled-out fraud
replicas each own a partition and keep per-subscriber state consistent.
